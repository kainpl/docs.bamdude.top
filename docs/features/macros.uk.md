# Макроси

Макроси -- це невеликі переюзабельні автоматизації, які спрацьовують у конкретні моменти життєвого циклу друку. BamDude підтримує два види макросів: класичні **G-code макроси** (принтер виконує власний G-code) та **MQTT-action макроси** (BamDude каже принтеру щось зробити через його MQTT control-канал — наприклад, перемкнути світло корпусу).

Використовуйте їх для механізмів заміни пластин, освітлення корпусу, керування корпусом або будь-якої іншої per-event автоматизації, якій не місце у start/end G-code слайсера.

---

## :material-code-braces: Типи дій

`action_type` макроса вирішує, що відбувається при спрацюванні події.

=== "G-code"

    Надсилає G-code-фрагмент через MQTT (`gcode_line`), загорнутий
    маркерами `M1002 gcode_claim_action`. Принтер ACK-ає фрагмент, а
    BamDude чекає, поки `stg_cur` принтера повернеться в idle, перш
    ніж рапортувати про завершення макроса.

    **Сценарії**

    - Леволінг столу, власні park-позиції, vibration calibration
    - Операції swap mode (виштовхнути + home + підготувати наступну плиту)
    - Будь-що, що ви б поклали у start/end G-code слайсера, але хочете,
      щоб оркестрував це BamDude

    **Обмеження**

    - G-code на `print_started` *б'ється* з самим друком — прошивка вже
      виконує власну стартову послідовність. Не робіть цього, якщо не
      розумієте точно, навіщо.
    - Довгий G-code на `swap_mode_change_table` -- ок; він виконується,
      коли принтер у idle між плитами.

=== "MQTT-action"

    Викликає іменовану MQTT-команду з каталогу BamDude замість того, щоб
    надсилати G-code. **Fire-and-forget** -- принтер не ACK-ає, тож
    BamDude нічого не чекає.

    **Команди, що зараз постачаються**

    | ID | Ефект |
    |----|-------|
    | `chamber_light_on` | Увімкнути світло корпусу |
    | `chamber_light_off` | Вимкнути світло корпусу |

    **Сценарії**

    - Авто-світло-увімкнено на старті друку, авто-світло-вимкнено на завершенні
    - Зовнішні автоматизації, яким не потрібен G-code-пайплайн

    Каталог живе в `core/mqtt_macro_actions.py` і доступний фронтенду
    через `/macros/meta`. Нові іменовані команди додаються туди в міру
    появи сценаріїв.

!!! info "Сумісність"
    G-code макроси з версій до 0.4.0 продовжують працювати без змін. Нові
    макроси за замовчуванням G-code, поки ви явно не виберете
    **MQTT-action** у редакторі.

## :material-lightning-bolt: Події

Кожен макрос прив'язаний рівно до однієї події. Обидва типи дій підтримують усі події нижче, але практичні застосування різні.

| Подія | Коли спрацьовує | Гарний фіт |
|-------|----------------|------------|
| **`print_started`** | Коли `gcode_state` переходить у `RUNNING` (через `on_print_start`) | MQTT-action (наприклад, `chamber_light_on`). G-code тут небезпечний. |
| **`print_finished`** | Коли друк досягає термінального стану (`FINISH`, `FAILED`, або `IDLE`-aborted), через `on_print_complete` | Симетрично до `print_started` -- паруйте з MQTT-action для чистої автоматизації lights-off. |
| **`swap_mode_start`** | Перед стартом друку, коли диспатч знає, що йдеться зі swap-mode профілем | G-code, що готує swap-механізм. |
| **`swap_mode_change_table`** | Після завершення друку, перед тим як черга візьме наступний елемент | G-code, що фізично змінює плити. Див. [Swap Mode](swap-mode.md). |

!!! tip "`print_finished` покриває всі термінальні стани"
    Незалежно від того, чи друк завершився чисто, впав, чи був перерваний
    з екрана принтера, `print_finished` спрацьовує один раз. Використовуйте
    його, коли поведінка має застосовуватися незалежно від результату
    (вимкнути світло, повідомити зовнішні системи тощо).

## :material-filter: Поля фільтрів

Обидва типи дій ділять той самий набір фільтрів. Макрос спрацьовує, лише коли **усі** фільтри збігаються.

| Поле | Поведінка |
|------|-----------|
| `enabled` | Глобальний перемикач. Вимкнені макроси пропускаються без подальшої оцінки. |
| `printer_models` | JSON-масив model-кодів (наприклад, `["A1 Mini", "X1 Carbon"]`) або `["*"]` для всіх моделей. |
| `swap_mode_only` | Спрацьовує тільки коли swap mode на принтері увімкнено. Прихований в UI для не-swap подій. |
| `swap_profile` | Спрацьовує тільки коли вибраний swap-профіль принтера збігається зі значенням (наприклад, `a1mini_v1`). Дозволяє кільком варіантам swap-mode G-code співіснувати. |
| `delay_seconds` | 0–3600. Відкласти дію на N секунд після тригера. 0 = негайно. |

### Чому `delay_seconds` важливий

Деякі події спрацьовують *якраз* перед тим, як принтер опиниться у видимому стані, на який ви очікуєте. Chamber-light-on на `print_started` виглядає передчасним на деяких моделях, бо firmware-side стартова послідовність (нагрів, purge) ще не закінчилась. Затримка 10–30 с уникає блимання без потреби писати власну state-машину.

## :material-pencil: Редагування макросів

1. Перейдіть до **Settings → Macros**.
2. Виберіть або створіть макрос. Виберіть **Action Type** (G-code або MQTT-action).
3. Виберіть **Event**.
4. Виставте поля фільтрів (моделі принтерів, swap mode, профіль, delay).
5. Для G-code макросів вбийте сніпет у редактор. Для MQTT-action виберіть команду з dropdown.
6. Save.

!!! warning "Спочатку перевірте G-code на принтері"
    Поганий G-code може пошкодити ваш принтер. Запускайте нові сніпети
    вручну, перш ніж прив'язувати їх до події.

---

## :material-cog-play: Як виконуються макроси

Макроси виконуються як **fire-and-forget asyncio таски**. Повільне відправлення G-code, довгий delay або навіть мережева заминка ніколи не блокують навколишню оркестрацію -- `on_print_start` повертає керування одразу, а макрос спрацьовує у фоні.

Для G-code макросів диспатчер додатково чекає на колбек `on_macro_complete` (`stg_cur` принтера повертається в idle) перш ніж переходити до *наступного* G-code макроса в тому ж event-ланцюзі. MQTT-action макроси повністю fire-and-forget; на них нічого не чекає.

## :material-history: Заміна старого прапора `auto_light_off`

До 0.4.0 у BamDude був boolean `auto_light_off` на кожному принтері. Його прибрано в міграції `m021`, бо макроси роблять ту саму роботу краще -- з контролем delay, симетрією on/off, націленням на конкретні моделі та фільтрами per-swap-profile.

Щоб відтворити стару поведінку:

=== "Світло геть, коли друк завершено"

    | Поле | Значення |
    |------|----------|
    | Action type | MQTT-action |
    | Event | `print_finished` |
    | Command | `chamber_light_off` |
    | `delay_seconds` | `0` |

=== "Світло на старті друку"

    | Поле | Значення |
    |------|----------|
    | Action type | MQTT-action |
    | Event | `print_started` |
    | Command | `chamber_light_on` |
    | `delay_seconds` | `10` (нехай нагрів спершу пройде) |

Паруйте обидва для повних циклів автоматичного освітлення. Додайте `swap_mode_only=true`, якщо хочете, щоб світло цикл оподібнялося лише на swap-mode прогонах.

---

## :material-swap-horizontal: Інтеграція зі Swap Mode

Swap-mode друки спираються на G-code макроси, прив'язані до `swap_mode_start` та `swap_mode_change_table`. Див. [Swap Mode](swap-mode.md) для повного життєвого циклу, restart-стійкого відстеження подій та порядку диспатчу на multi-printer фермах.

---

## :material-lightbulb: Поради

!!! tip "Паруйте `print_started` із `print_finished`"
    Використовуйте по одному MQTT-action макросу на кожну подію для
    чистої симетричної автоматизації (світло, вентилятори, зовнішні реле).

!!! tip "Використовуйте `delay_seconds` для chamber-light-on"
    Затримка 10–30 с приховує фазу нагріву від камери корпусу та
    уникає вигляду "передчасного увімкнення світла" на H2/X1.

!!! tip "Поєднуйте з розумними розетками"
    Для повних циклів виштовхування + охолодження + вимкнення живлення
    паруйте end-event макроси з авто-вимкненням [розумних розеток](smart-plugs.md).
