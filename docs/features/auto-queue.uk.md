# Авто-черга (Auto-Queue Routing)

Авто-черга — це **шар-маршрутизатор поверх черг окремих принтерів**. Кидаєш у неї друк без вказівки конкретного принтера — планувальник сам обирає придатний вільний принтер (за моделлю + типом і кольором філаменту), копіює елемент у чергу того принтера, а далі вже відпрацьовує звична per-printer диспетчеризація.

Зʼявилася в **0.4.2**.

---

## :material-router-network: Як це працює

```
┌────────────────┐   є придатний вільний принтер?
│ AutoQueueItem  │────────┐
│ (target_model, │        │ так → копіюється в print_queue принтера
│  filaments,    │        │       auto-рядок стає "assigned"
│  colors)       │        │
└────────────────┘        ▼
                     PrintScheduler підхоплює його
                     протягом ~1 тіку (background_dispatch)
```

Фоновий цикл (`AutoQueueScheduler`, стартує з `main.py` lifespan) прокидається кожні **30 секунд** і на кожному тіку:

1. **Знімає список зайнятих принтерів** — усе що зараз `status='printing'` в `print_queue` виключається з цього раунду.
2. **Читає pending авто-чергові рядки** в порядку SJF (Shortest Job First) + `been_jumped`, якщо ввімкнено налаштування **Queue Shortest First**, інакше — за `position`.
3. **Для кожного елемента викликає `find_eligible_printer`** — шукає вільний принтер що відповідає:
    - `target_model` елемента (наприклад `X1C`, `P1S`, `A1MINI`, `H2D`)
    - усім `required_filament_types` (витягнуті з 3MF, користувач може перевизначити)
    - вимогам по кольору (коли `force_color_match=true`)
    - опційному `target_location` (тег кімната/стелаж), якщо ти групуєш принтери за локацією.
4. **Якщо є збіг** — елемент копіюється в `print_queue` цього принтера з AMS-мапінгом, обчисленим зі поточного стану котушок принтера. Auto-рядок переходить у `status='assigned'` і отримує посилання на створений per-printer елемент.
5. **Якщо збігу немає** — оновлюється `waiting_reason`, щоб черговий UI міг пояснити, *чому* елемент стоїть (наприклад `"всі P1S зайняті"`, `"треба PETG (червоний), у жодному принтері не заряджений"`).

Щойно рядок осідає в per-printer черзі — спрацьовує **звичний потік диспетчеризації**: гейт plate-clear, staggered start, swap-макроси, сушка, усе як завжди. Маршрутизатор нічого не оминає.

## :material-sort-numeric-ascending: SJF + захист від голодування

Коли увімкнено **Queue Shortest First** (Налаштування → Друк → Queue & Scheduling → Auto-Queue Routing → Queue Shortest First), pending-рядки сортуються:

```
ORDER BY been_jumped DESC,
         estimated_print_time_seconds ASC,
         position ASC
```

Sticky-біт `been_jumped` запобігає голодуванню: щоразу як SJF проштовхує коротший друк *повз* довший, довший отримує `been_jumped=True` і випливає на верх наступного раунду незалежно від часу друку. Так 14-годинний друк не стоятиме безкінечно за потоком 30-хвилинних завдань.

Якщо тогл вимкнено — елементи диспатчаться в порядку FIFO (за `position`).

## :material-cog: Налаштування

**Налаштування → Друк → Queue & Scheduling → Auto-Queue Routing**:

| Налаштування | Що робить |
|--------------|-----------|
| **Queue Shortest First** | Вмикає SJF + захист від голодування. За замовчуванням: вимкнено (FIFO). |

Сам маршрутизатор **завжди увімкнений** — головного перемикача немає. Якщо в авто-черзі немає елементів — планувальник просто 30-секундний no-op.

## :material-plus-circle: Як додати в авто-чергу

Чотири способи кинути друк в авто-чергу:

### 1. Print Modal — тогл "Auto"

В Print Modal (з архівів і з бібліотеки) є тогл **Specific / Auto**. Вибираєш **Auto** — модалка показує:

| Поле | Примітки |
|------|----------|
| Target Model | Заповнюється автоматично з `sliced_for_model` 3MF. За замовчуванням "any" — лиши порожнім, щоб будь-який сумісний принтер міг підхопити. |
| Target Location | Опційний тег кімната/стелаж, якщо ти позначив принтери. |
| Force Color Match | Коли увімкнено, перевірка вимагає принтер що має кожен філамент *і потрібний колір*. За замовчуванням вимкнено — збігається лише за типом. |

Submit — і в БД зʼявляється `AutoQueueItem`.

### 2. Режим `auto_queue` віртуального принтера

Слайсер "Send to Printer" → ВП отримує файл → архівується → потрапляє в авто-чергу. Деталі UI див. [Віртуальний принтер → auto_queue](virtual-printer.md#auto_queue).

Це повністю hands-off шлях "наслайсив і забув": слайсер не знає на якому принтері друкуватиметься, і оператор теж не знає, поки маршрутизатор не вирішить.

### 3. Drag-and-drop на панель Auto-Queue

Перетягни нарізаний файл (`.gcode` / `.gcode.3mf`) куди завгодно над **панеллю Auto-Queue** на верху сторінки Queue. Файл аплоудиться у корінь бібліотеки, потім відкривається той самий Print Modal, заблокований у режимі **Auto** (без тогла specific/auto, без printer picker — лише constraint-и auto: target model / location / force-color). Submit — і створюється `AutoQueueItem`.

Без перевірки сумісності моделі на дропі (на відміну від drop-у на картку per-printer черги) — auto-router сам фільтрує кандидатів за `sliced_for_model` під час диспатчу, тож несумісний файл просто висить у панелі з `waiting_reason`, поки не з'явиться придатний принтер або оператор не скасує.

Захищено правом `queue:create`. Панель рендериться завжди, навіть порожня — drop-target постійно доступний; підказка empty-state нагадує про drag-drop при першому відкритті.

### 4. REST API

```http
POST /api/v1/auto-queue/
{
  "library_file_id": 42,
  "target_model": "P1S",
  "force_color_match": false
}
```

Повна схема — у [API reference](../reference/api.md). Quantity > 1 створює N рядків одним викликом (та ж семантика `batch_id`, що й у `print_queue`).

## :material-monitor-dashboard: Панель AutoQueuePanel на Queue dashboard

На сторінці Queue є **панель Auto-Queue** над картками per-printer черг. **Рендериться завжди**, тож drop-target постійно доступний; коли pending-айтемів немає, панель згортається до однорядкової підказки про drag-drop. Інакше показує pending авто-чергові елементи:

- мініатюра, назва, плейт, target model
- розрахунковий час друку
- причина очікування, якщо немає придатного принтера
- inline-кнопки: edit (target model / location / force-color), cancel

Щойно елемент призначається на принтер — він зникає з панелі і зʼявляється в чергу того принтера з невеликим бейджем "auto-assigned".

## :material-filter-variant: Правила придатності

Гелпер `find_eligible_printer` вважає принтер придатним, коли виконується **все** з нижченаведеного:

| Перевірка | Деталі |
|-----------|--------|
| **Простоює** | Не `status='printing'`, не `paused`, не `error`, не зайнятий інакше. |
| **Збіг моделі** | Якщо вказано `target_model`, код моделі принтера має дорівнювати йому. |
| **Збіг локації** | Якщо вказано `target_location`, тег локації принтера має дорівнювати йому. |
| **Типи філаменту** | Кожен потрібний тип філаменту має бути в якомусь завантаженому слоті (AMS або external spool). |
| **Збіг кольору** | При `force_color_match=true` hex кольору теж має збігатися для кожного філаменту. |
| **Filament-overrides** | Будь-яка per-print заміна (наприклад "використати PLA Tough замість PLA") застосовується перед перевіркою завантажених слотів. |

Tie-breaker — коли придатних принтерів кілька:

1. **Найменший ужиток філаменту** (коли увімкнено налаштування `prefer_lowest_filament`, за замовч. вимкнено). Бере принтер у якого AMS-слоти мають найменший залишковий грамаж по потрібним філаментам — лишає "свіжі" котушки на серйозніші завдання.
2. Інакше — принтер з найменшим id.

## :material-clipboard-text: AMS-мапінг на момент призначення

Коли маршрутизатор копіює авто-чергу в per-printer `print_queue`, він **обчислює AMS-мапінг тут і зараз** з поточного стану котушок принтера — не на момент сабміту. Тож якщо ти переставив котушку між сабмітом і диспатчем — призначений мапінг відображає нову котушку. Початкові `filament_overrides` з auto-рядка застосовуються першими, потім авто-мапінг заповнює решту слотів.

## :material-link-variant-off: Семантика cancel / edit

| Дія | Ефект |
|-----|-------|
| Cancel pending авто-черги | Рядок видалений. Жоден принтер його не бачив. |
| Cancel assigned авто-черги | Скасовується **per-printer елемент**, якого створив маршрутизатор. Auto-рядок лишається у `assigned` для аудиту. |
| Edit `target_model` / location / force-color | Дозволено лише поки `pending`. Після призначення — редагуй per-printer елемент. |

## :material-lightbulb: Коли користуватися

| Сценарій | Рекомендація |
|----------|--------------|
| Один принтер | Пропусти авто-чергу, користуйся звичною per-printer чергою — без overhead на маршрутизацію, простіший UI. |
| 2-3 принтери однієї моделі | Авто-черга — золотий варіант для load-balancing: кидаєш завдання, планувальник розгрібає на той що звільнився першим. |
| Змішана ферма по моделях | Авто-черга з явним `target_model` per job — той самий load-balancing у межах моделі, без кросу між моделями. |
| Кольоро-чутливі завдання (логотипи, вивіски) | Увімкни `force_color_match` — завдання не диспатчиться на принтер з не тим кольором. |
| Hands-off потік від слайсера | ВП у режимі `auto_queue` + авто-черга → повністю unattended slice → print pipeline. |

## :material-history: Перехід зі звичайної черги

Якщо ти користувався опцією **"будь-який принтер моделі X"** у пікері per-printer черги — це legacy single-tier router. Він досі працює, але авто-черга його перевершує:

| Можливість | Legacy "any of model X" | Авто-черга |
|------------|--------------------------|------------|
| Бере вільний принтер вказаної моделі | так | так |
| Збіг типу філаменту | ні — оператор сам перевіряє | так |
| Збіг кольору | ні | опційно (`force_color_match`) |
| SJF + захист від голодування | ні | так |
| Фільтр локації | ні | так |
| Видима панель "очікування" з причинами | ні | так |
| Інтеграція з ВП | лише proxy | окремий режим `auto_queue` |

Авто-конверсії немає — існуючі per-printer елементи продовжують працювати. Користуйся авто-чергою для нових завдань, де рішення про маршрутизацію можна відкласти.
