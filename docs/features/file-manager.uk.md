# Файловий менеджер

Переглядайте, завантажуйте та керуйте файлами у локальній бібліотеці BamDude. Друкуйте напряму або додавайте до черги.

---

## :material-folder: Огляд

Файловий менеджер дозволяє:

- **Переглядати** файли у локальній бібліотеці
- **Монтувати зовнішні папки** з NAS, USB або мережевих ресурсів
- **Завантажувати** файли, включаючи ZIP-архіви
- **Друкувати напряму** на будь-якому принтері
- **Додавати до черги** нарізані файли для подальшого друку
- **Перейменовувати** та **видаляти** файли і папки

---

## :material-printer: Прямий друк

1. Знайдіть нарізаний файл (`.gcode` або `.gcode.3mf`)
2. Натисніть іконку принтера або клацніть правою кнопкою для контекстного меню
3. Виберіть **Print**
4. Оберіть принтер(и), налаштуйте відповідність філаментів, задайте параметри друку
5. Натисніть **Print** для запуску

### Додавання до черги

Додавайте нарізані файли до черги для подальшого друку без попереднього створення архівів. Архіви створюються автоматично при запуску друку.

---

## :material-folder-zip: Завантаження ZIP-файлів

Завантажуйте ZIP-архіви для розпакування вмісту у бібліотеку:

1. Натисніть **Upload** та виберіть `.zip` файл
2. Оберіть, чи зберігати структуру папок
3. Натисніть **Extract**

---

## :material-cursor-default-click: Drag-and-drop завантаження

Перетягніть один або кілька файлів куди завгодно над **зоною файлів** (правою панеллю з тулбаром і сіткою/списком файлів) — модалка завантаження відкриється з вже доданими файлами. Та сама pipeline, що й picker / drop-зона у самій модалці, тож опції ZIP залишаються видимі і можна скасувати до фактичного аплоуду. Поточна вибрана тека зберігається як призначення.

Page-level drop захищено правом `library:upload` — у в'юверів без нього оверлей не з'являється і drop ігнорується.

---

## :material-database: Як зберігаються файли

Кожен файл бібліотеки — це рядок у таблиці `library_files`. Рядок несе:

- **Hash-дедуп** — завантажені файли SHA-256-яться і зіставляються з наявними рядками; ідентичний повторний upload повертає існуючий запис, замість створення дубля копії на диску.
- **Мініатюри** — витягуються з `Metadata/plate_*.png` всередині 3MF при завантаженні (без on-the-fly екстракції). Повторний upload або "reparse" тригерять нову екстракцію.
- **STL thumbnail render** — STL-завантаження (`.stl`, `.zip` зі STL всередині) отримують мініатюру, рендерену на upload через вбудований растеризатор, тож на картці видно справжню деталь, а не generic placeholder.
- **`print_count` + `last_printed_at`** — лічильники використання, які підтримує dispatch; видно при наведенні на картку файла, використовуються режимами сортування. Заповнюються ретроактивно при апгрейді через міграцію `m014`.
- **JSON-колонка `file_metadata`** — зберігає розпарсені metadata слайсера: ваги філаменту по котушках, кількість об'єктів, цільова модель принтера в слайсері, плюс прапорці `gcode_label_objects` / `exclude_object` з `Metadata/project_settings.config` вихідного 3MF (витягуються в 0.4.1, заповнюються міграцією `m022`). Прапорці label-objects є воротами для кнопки **skip-objects** на сторінці принтера під час друку — обидва мають бути `true`, щоб кнопка засвітилася. Bambu Studio вмикає обидва за замовчуванням; OrcaSlicer постачає обидва вимкненими (див. [Troubleshooting](../reference/troubleshooting.uk.md) для чек-листа на стороні слайсера).
- **`is_multi_plate` + per-plate cache `plates[]` (m023)** — для multi-plate sliced 3MF (один `.gcode.3mf` з кількома `Metadata/plate_N.gcode` всередині) BamDude передекстрактує повний per-plate breakdown — мініатюру, час друку, вагу пластика, кількість об'єктів, filament-стек, label-object прапорці — у той самий `file_metadata` JSON. Список файлів повертає це без перевідкривання 3MF на кожен запит.
- **Прапорець `swap_compatible`** — визначається за маркером `.swap.` або `.swaps.` в імені файла, наприклад `MyPart.swap.gcode.3mf` або `Tray.swaps.3mf`. Маркер має бути **через крапку**, не через підкреслення — `MyPart_swap.gcode.3mf` НЕ позначиться. Swap-сумісні файли окремо виносяться в swap-mode picker.
- **Композитна колонка `file_tags` (m036 / m037)** — неупорядкований JSON-список тегів-ідентичностей керує і баджами, і чіп-фільтром на тулбарі. Чотири семантичні групи: **format** (`gcode` / `3mf` / `stl` / `obj` / `step` — сліцений `.gcode.3mf` тримає композитну пару `gcode + 3mf`, тож візуальне розрізнення зберігається після колапсу `file_type`), **readiness** (взаємовиключне: `sliced` для слайсер-аутпуту, `project` для нерізаного `.3mf`-пакета, `geometry` для голої сітки / CAD-джерела — один тоглер для "що ще треба нарізати?"), **modifiers** (`swap` / `multiplate`), **provenance** (`makerworld`). Фронтенд `sortTagsForDisplay` проектує на явний пріоритет так, що ряд читається справа-наліво: format → readiness → modifiers → provenance.

## :material-tag-multiple: Чіп-фільтр тегів

Тулбар несе ряд чіпів над списком файлів. Кожен чіп — тег із `file_tags`; клік перемикає його. Вибрані чіпи AND-фільтрують список, тож наприклад `multiplate + sliced` повертає тільки мульти-плейт-сліцені файли. Вибір зберігається в localStorage, тож фільтр переживає перезавантаження. Рендеряться тільки чіпи для тегів, що реально присутні у завантаженому списку — інсталяції без певних provenance-джерел бачать тісніший ряд.

## :material-eye-outline: 3D / G-code в'ювер

Файли бібліотеки шерять той самий `<ModelViewerModal>`, що й архіви, з двома бібліотечно-специфічними штрихами:

- **Видимість вкладок з `file_tags`, а не з розширення файлу.** Сліцений `.gcode.3mf` (у якого обидва розширення) показує тільки вкладку G-code — його вбудована сітка вже растерізована в gcode-лінії, рендерити її ще раз під "3D Model" — дублікат. Нерізаний `project` 3MF або голий `geometry` mesh показує тільки 3D-вкладку. Вкладки питаються через `GET /library/files/{id}/capabilities` (дзеркало давнього архівного ендпоінта).
- **Per-plate G-code picker для мульти-плейт бібліотечних файлів.** Файли бібліотеки — для browse, тож G-code-вкладка отримує picker плит, що пере-ключає URL gcode-preview, коли перемикаєш плиту. Архіви тримають single-plate-поведінку, бо вони — запис одного конкретного друку.
- **Вайрфрейм друкарського об'єму** — 3D-в'ювер малює напівпрозорий бокс, що відповідає принтеру, для якого файл нарізано (читається з `printer_settings`). G-code preview уже малював схожий бокс; тепер однаковий візуальний натяк на обох вкладках.

## :material-view-gallery: Per-plate gallery (multi-plate 3MF)

Sliced 3MF з більш ніж однією плитою рендеряться як per-plate gallery на картці файла:

- Вертикальна paginator-стрічка ліворуч — одна кнопка на плиту, кожна з точкою стану виділення.
- Велика картка праворуч з мініатюрою цієї плити, назвою, часом друку, загальною вагою, кількістю об'єктів і per-filament breakdown (color swatch + тип + грами).
- Виділення (які плити друкуємо) і навігація (картка якої плити зараз видна) розв'язані — можна гортати плити, не міняючи селекшен.

При диспатчі можна вибрати одну плиту, кілька або всі — кожна вибрана плита стає окремим queue-елементом / архівом з індексом плити, записаним у рядок.

Single-plate файли gallery не рендерять — наявна головна мініатюра і так усе показує.

## :material-link-variant: Лінки на проєкти та папки

- **Лінк на папку** -- лінкування папки до проєкту проставляє `project_id` кожному файлу всередині, і будь-який файл, переміщений у цю папку пізніше, успадковує лінк.
- **Лінк на файл** -- кожен рядок файла також має власну кнопку `Link2`, щоб приєднати його до проєкту незалежно від папки.
- **Plan items проєкту** (m016) -- сторінка проєкту рендерить плаский plan-список з копіями/порядком/підсумками; рядки з'являються автоматично, коли файли / папки лінкуються до проєкту, і per-row підсумки (філамент, час, вартість) живлять загальні підсумки проєкту.

---

## :material-folder-network: Монтування зовнішніх папок

Монтуйте директорії хоста (NAS, USB-накопичувачі) у файловий менеджер без копіювання файлів:

1. Підключіть директорію через bind-mount у Docker
2. Натисніть **Link External** на панелі інструментів
3. Введіть відображувану назву та шлях у контейнері
4. Файли індексуються та з'являються негайно

---

## :material-lightbulb: Поради

!!! tip "Підтримка кількох принтерів"
    Виберіть кілька принтерів, щоб надіслати один файл на всю ферму одночасно.

!!! tip "Позначки файлів"
    Шукайте позначки "sliced", щоб визначити файли, готові до друку.
